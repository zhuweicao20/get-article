# 论文图文公众号工作流 Skill：参考推文风格+短背景版

## 适用场景
当用户上传科研论文 PDF，并要求生成“图文并茂公众号推文 / 中文图文解读 / HTML / PDF / 打包结果”时，使用本 skill。

本 skill 配合 `paper_wechat_workflow_reference_push_style_v5` 使用。工作流负责 PDF 文本提取、Figure caption 定位、图片裁剪、Markdown/HTML/PDF/zip 导出；AI 负责理解论文、筛选重点、改写为参考公众号语言、检查图片和修正输出。

## 参考推文风格
用户提供的参考推文主题为 “MOF玻璃，最新Nature Chemistry！”；其主要排版和叙事特点如下：

```text
黑色标题：MOF玻璃，最新Nature Chemistry！
公众号来源行：高分子科学前沿  时间  地区  听全文
正文开头：一段加粗的研究亮点句
主体：连续叙事，不分很多大卡片标题
重点：作者/团队/材料体系/结论用蓝色或橙色加粗强调
图片：论文原图接近正文全宽插入
图注：直接跟在图片下方，长图注左对齐/两端对齐
图后：紧接 1–2 段中文解释
结尾：总而言之，概括创新和启发
```

广告图、推广服务、推荐阅读、留言区、游戏卡片等不属于正文内容，默认忽略，不自动生成。

## 与上一版的关键差别

上一版使用“全文速览 / 背景介绍 / 图文解析 / 总结与启发”的蓝色装饰小标题。参考推文不是这种卡片式小标题，而是连续文章式排版。因此本版要求：

- 不再默认使用蓝色居中小标题卡片；
- 不把文章切成很多明显栏目；
- 用加粗首句和自然过渡段完成结构引导；
- 图片、图注、解读段落连续排列；
- 重点词句可用蓝色/橙色强调。

## 总目标
将科研论文处理成一套可交付的中文图文内容包：

- `article.md` / `article_rich.md`：公众号图文稿；
- `article.html` / `article_rich.html`：浏览器预览版，尽量贴近参考推文排版；
- `article.pdf`：若环境支持 WeasyPrint，则导出整合 PDF；
- `article.json`：结构化结果；
- `selected_charts.json`：入选图表和说明；
- `review_report.json`：质量检查报告；
- `images/`：从论文中提取或裁剪出的图；
- `outputs.zip`：最终打包文件。

## 输入
用户可能上传：

1. 论文 PDF；
2. 工作流 zip，例如 `wechat_style_workflow_and_skill_v3.zip`；
3. skill zip 或 `SKILL.md`；
4. 参考推文截图/PDF；
5. 期望字数、风格、平台，例如“公众号”“小红书”“组会精读 PDF”。

默认输出：

- 中文公众号图文稿；
- 约 1500–2200 字；
- 在正文开头加入一小段该领域背景，控制在 80–150 字，不写成长综述；
- 插入 3–6 张关键 Figure；
- 语言通俗但不夸大结论；
- 保留原文题目、DOI、期刊、核心数据；
- 忽略广告模块。

## 执行流程

### 1. 解压与检查工作流
如果用户上传了工作流 zip：

```bash
unzip -q wechat_style_workflow_and_skill_v3.zip -d workflow_dir
cd workflow_dir/workflow/*
find . -maxdepth 3 -type f
```

检查是否至少包含：

```text
src/main.py
src/tools/pdf_tool.py
src/tools/figure_extractor.py
src/tools/article_writer.py
src/tools/exporter.py
requirements.txt
README.md
```

若缺少关键文件，应说明缺失并尽量补齐。

### 2. 安装依赖
优先使用当前环境已有依赖。如果缺少 PyMuPDF / Pillow / WeasyPrint / Markdown 等，使用：

```bash
pip install -r requirements.txt
```

如果无法安装依赖，应改用已有 PDF 工具或 Python 库兜底。WeasyPrint 缺失时，不要中断工作流，应写出 `pdf_export_warning.txt` 并继续输出 HTML/Markdown/zip。

### 3. 运行基础工作流
对用户上传 PDF 执行：

```bash
python src/main.py --pdf /path/to/paper.pdf --out /path/to/outputs --word-count 1800 --max-figures 8
```

确认输出：

```bash
find /path/to/outputs -maxdepth 3 -type f
```

至少应有：

```text
article.md
article.html
article.json
selected_charts.json
review_report.json
outputs.zip
images/
```

### 4. 检查图片裁剪质量
必须检查 `outputs/images/`，不能默认相信工作流结果。

重点检查：

- 是否把正文大段文字截进去了；
- 是否漏掉 Figure 的上半部分或右半部分；
- 是否只截到 caption 没截到图；
- 是否重复截同一张图；
- 是否图片过糊、过小、空白太多；
- 是否 Figure 编号和正文解释对应。

如果图片裁剪错误，按优先级修复：

1. 使用 PDF image block 的真实 bbox 裁剪；
2. 使用 Figure caption 作为锚点，裁 caption 上方完整图区；
3. 对跨栏图，扩大横向区域到整页宽度；
4. 对整页大图，直接渲染该页后人工裁剪图区；
5. 对跨页图，分开裁剪并在正文中说明。

### 5. 论文内容理解
必须至少提取以下信息：

- 论文题目；
- 期刊和年份；
- DOI；
- 第一作者、通讯作者、通讯单位；
- 研究对象 / 材料体系；
- 研究问题；
- 关键创新点；
- 关键实验或计算方法；
- 3–5 个核心结果；
- 最重要的定量数据；
- 论文结论；
- 局限性或不能夸大的地方。

禁止只根据摘要生成全文。应结合摘要、结果、图注和结论。

### 6. 图表选择原则
优先选择能支撑文章主线的图，而不是简单按 Figure 顺序全部插入。

优先级：

1. 论文首页图 / 研究对象图 / 设计示意图；
2. 核心结构表征图；
3. 热学、电学、吸附、催化或电池等核心性能图；
4. 机理验证图；
5. 应用或稳定性验证图。

每张图下面必须写中文解释，解释应包括：

- 这张图想证明什么；
- 读者应该看哪几个 panel；
- 图中数据和论文结论的关系；
- 不要夸大的边界。

### 7. 公众号文章结构
默认结构改为参考推文风格。Markdown 结构建议：

```markdown
# MOF玻璃，最新Nature Chemistry！

高分子科学前沿　2026年5月4日 18:48　广东　听全文

第一作者：...
通讯作者：...
通讯单位：...
论文 DOI：...
期刊信息：...

**原文题目：...**

一小段领域背景。只说明这个方向为什么重要、目前常见难点是什么，控制在 80–150 字。

正文导语。说明研究问题、材料体系和创新点。

**针对这一挑战，某某团队成功开发出一种……** 这里用蓝色/橙色加粗突出作者、材料和关键结论。

![图1](images/figure_1.png)

图 1｜原文图注中文整理……

图 1 解读段落……

![图2](images/figure_2.png)

图 2｜原文图注中文整理……

图 2 解读段落……

总而言之，……
```

标题风格：

- 推荐：“MOF玻璃，最新Nature Chemistry！”、“固态电池，最新JACS！”、“催化材料，最新Nature Energy！”；
- 不要标题党，不要写“颠覆”“炸裂”“彻底解决”；
- 可以突出材料体系和期刊。

语言要求：

- 中文自然，不要翻译腔；
- 适合科研公众号读者；
- 领域背景只做轻量铺垫，不要堆历史、不写综述；
- 不要写成论文摘要；
- 不要过度口语化；
- 核心数据必须准确；
- 每段尽量短，方便手机阅读；
- 对不确定内容用“表明 / 提示 / 可能 / 需要进一步验证”，不要过度断言。

### 8. HTML 排版要求
`article.html` 应尽量模拟参考公众号文章：

- 最大宽度约 720px；
- 标题黑色粗体，约 22px；
- 公众号来源/时间/地区为灰色小字；
- 正文字体：微软雅黑 / PingFang SC / 系统中文字体；
- 正文 16px，行距约 1.8；
- 重点句加粗，重要结论可用橙色，关键词和作者信息可用蓝色；
- 图片宽度 100%，不加明显卡片阴影；
- 图注 14px，颜色 #333，左对齐/两端对齐；
- 不插入广告图和推广模块。

### 9. 版权与引用原则
处理 Nature、Science、JACS、ACS、Springer Nature、Wiley、Elsevier 等论文时：

- 不要整篇搬运原文；
- 不要大段直译；
- 图表应仅用于个人研究整理或草稿预览；
- 正式公开发布前，建议用户确认图表版权或使用自己重绘图；
- 文章中应保留原文题目、期刊、DOI 或链接。

### 10. 质量检查
导出前必须检查：

- 文章是否忠实于论文；
- Figure 编号是否对应；
- 图片是否能打开；
- Markdown 图片路径是否正确；
- HTML 是否能显示图片；
- 是否没有生成广告、推广、留言区；
- 重点句颜色是否适度，不能满篇彩色；
- 最后一段是否像参考推文一样用“总而言之”或类似表达收束。

### 11. 常见问题处理

如果用户说“图没有了 / 图片不显示”：

- 检查 `images/` 是否存在；
- 检查 Markdown 中是否为相对路径 `images/figure_1.png`；
- 检查 HTML 的 `base_url` 是否指向输出目录；
- 打包时必须把 `images/` 放进 `outputs.zip`。

如果用户说“复制到公众号是代码”：

- 说明 `article.html` 是网页预览，公众号后台通常不能直接粘 HTML 源码；
- 让用户在浏览器打开 `article.html`，用鼠标框选网页内容复制，再粘贴到公众号编辑器；
- 或使用支持 HTML 导入的编辑器/插件；
- 图片需要一并上传到公众号素材库或通过编辑器导入。

如果用户说“参考推文里的广告先不管”：

- 明确执行为：广告图、推广服务、留言区都不做；
- 只模仿正文排版、标题、图片和图注风格。
